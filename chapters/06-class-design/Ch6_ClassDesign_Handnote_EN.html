SECTION 01

Understanding Inheritance

Inheritance lets a subclass (child) automatically include certain members — fields and methods — of a superclass (parent). You declare it with the extends keyword. The parent must not be final.

What gets inherited

Access levelInherited by subclass?
publicYes — always
protectedYes — always
package (default)Only if same package
privateNo — never inherited
 
public class BigCat {
    protected double size;
}
public class Jaguar extends BigCat {
    public Jaguar() { size = 10.2; }       // OK: size is protected, inherited
}
public class Spider {
    public void printDetails() {
        System.out.println(size);            // DOES NOT COMPILE — Spider doesn't inherit size
    }
}

Inheritance is transitive

If X extends Y and Y extends Z, then X is a descendant of Z. A direct subclass is the one that literally writes extends — so X is a direct subclass only of Y, not of Z.

⚠ Exam Trap

If you see a class marked final being extended, the code does not compile — instantly.

public class Mammal {}
public final class Rhinoceros extends Mammal {}
public class Clara extends Rhinoceros {}   // DOES NOT COMPILE — Rhinoceros is final

Single inheritance (Java)

  • A class has exactly one direct parent.
  • Multiple levels are allowed (X→Y→Z).
  • One parent may still have many children.

Multiple inheritance (not in Java)

  • A class with multiple direct parents — disallowed.
  • Causes ambiguity over which parent to inherit from.
  • Only exception: a class may implement many interfaces (Ch. 7).

Inheriting Object

Every class inherits from java.lang.Object. It is the only class with no parent. If your class doesn’t extend anything, the compiler silently adds extends java.lang.Object. So these two are equivalent:

public class Zoo { }
public class Zoo extends java.lang.Object { }   // equivalent
★ Remember

When you extend an existing class, Java does not add Object again — you already inherit it through the chain. Primitives (int, boolean) are not classes, so they don’t inherit Object; their wrapper classes do (via autoboxing). Methods like toString() and equals() live in Object, so every class has them.

SECTION 02

Creating Classes

Extending a class

// Animal.java
public class Animal {
    private int age;                  // private — not directly inherited
    protected String name;          // protected — inherited & directly accessible
    public int getAge() { return age; }
    public void setAge(int newAge) { age = newAge; }
}
// Lion.java
public class Lion extends Animal {
    protected void setProperties(int age, String n) { setAge(age); name = n; }
    public void roar() {
        System.out.print(name + ", age " + getAge() + ", says: Roar!");
    }
}

lion.setProperties(3,"kion") then lion.roar() prints kion, age 3, says: Roar!. age is private, reachable only via setAge()/getAge(); name is protected, so it’s directly accessible. Accessing age directly inside Lion would not compile.

Class access modifiers (top-level)

Top-level class rules

  • A .java file may have at most one public top-level class.
  • You may have as many package-access classes as you want, in any order — and you don’t even need a public one.
  • A top-level class can only be public or package — protected and private are illegal here (allowed only on nested classes, Ch. 7).
protected class ClownFish {}    // DOES NOT COMPILE
private class BlueTang {}       // DOES NOT COMPILE

The this reference

When a parameter shares a name with an instance variable, Java uses the most granular (narrowest) scope. Watch this:

public class Flamingo {
    private String color = null;
    public void setColor(String color) {
        color = color;                 // assigns parameter to itself — instance var untouched!
    }
}
// f.setColor("PINK"); System.out.print(f.color);  => null

Fix it with this.color = color; → prints PINK. The this reference points to the current instance and works in any instance method, constructor, or instance initializer — but not in a static method or static initializer (no implicit instance there).

TRACE — the Duck example → prints 0 2 white

  • length = this.length; — backwards! Assigns instance length (0) to the parameter; instance stays 0.
  • height = theHeight; — different names, no this needed; sets height to 2.
  • this.color = "white"; — fine; this. is legal even without a name clash.

The super reference

When a variable/method exists in both parent and child, the instance holds two copies. this searches current → inherited; super searches inherited only (it excludes members of the current class).

public class Reptile { protected int speed = 10; }
public class Crocodile extends Reptile {
    protected int speed = 20;
    public int getSpeed() { return speed; }        // => 20 (this.speed)
    // return super.speed;  would give => 10
}
⚠ Exam Trap — Insect / Beetle

This program does not compile because of one line:

class Insect { protected int numberOfLegs = 4; String label = "buggy"; }
public class Beetle extends Insect {
    protected int numberOfLegs = 6;
    short age = 3;
    public void printData() {
        System.out.println(this.label);    // buggy  (inherited)
        System.out.println(super.label);   // buggy  (inherited)
        System.out.println(this.age);      // 3      (current class)
        System.out.println(super.age);     // DOES NOT COMPILE — age not inherited
        System.out.println(numberOfLegs);   // 6      (narrowest scope = Beetle)
    }
}

this = current + inherited; super = inherited only. age lives only in Beetle, so super.age is illegal.

SECTION 03

Declaring Constructors

A constructor matches the class name exactly and has no return type (not even void). It runs when new creates an instance.

public class Bunny {
    public bunny() {}          // DOES NOT COMPILE — lowercase name = method, but no return type
    public void Bunny() {}     // legal METHOD (has return type) — NOT a constructor
}
⚠ Exam Trap

Constructor parameters may be any class, array, primitive, or generic — but never var: public Bonobo(var food){} does not compile.

The default constructor

Default no-arg constructor

  • If a class declares no constructors at all, the compiler inserts a public no-argument constructor.
  • It is generated at compile time — it never appears in the .java file, only in the .class.
  • If you declare any constructor (even a private one), no default is inserted.
ClassHas a constructor coded?Gets a default no-arg?
Rabbit1 {}NoYes
Rabbit2 with Rabbit2()YesNo
Rabbit3 with Rabbit3(boolean)YesNo
Rabbit4 with private Rabbit4()Yes (private)No

new Rabbit4() from another class does not compile — the constructor is private. A class with only private constructors can’t be instantiated externally and gets no default (useful for static-only utility classes).

Calling overloaded constructors with this()

Constructors can call one another, but only via this(...) — not by name and not with new (that would create a second, throwaway object).

public Hamster(int weight) {
    Hamster(weight, "brown");        // DOES NOT COMPILE — can't call a constructor like a method
}
public Hamster(int weight) {
    new Hamster(weight, "brown");    // compiles, but builds an EXTRA discarded object
}
public Hamster(int weight) {
    this(weight, "brown");          // CORRECT — same instance
}

this vs this()

  • this → a reference to the current instance.
  • this() → a call to another constructor in the same class.

super vs super()

  • super → references a parent member.
  • super() → calls a parent constructor.

Constructor rules to memorize

  1. A class can have many overloaded constructors, each with a distinct signature.
  2. The first line of every constructor is a call to this() or super().
  3. If this() / super() is present, it must be the first statement (so only one such call per constructor).
  4. If neither is written, the compiler inserts a no-arg super() as the first line.
  5. Java forbids cyclic constructor calls.
  6. super() always refers to the most direct parent.
⚠ Exam Trap — statement before this() 
public Hamster(int weight) {
    System.out.println("chew");      // a statement...
    this(weight, "brown");          // DOES NOT COMPILE — this() must be first
}

Comments don’t count as statements, so a comment before this() is fine. Cyclic calls also fail: Gopher(int){ this(5); } calling itself does not compile.

Calling parent constructors with super()

public class Zoo {
    public Zoo() {
        System.out.println("Zoo created");
        super();                       // DOES NOT COMPILE — super() must be first
    }
}

A child constructor may call any accessible parent constructor — it need not match its own signature. super() taking arguments still counts as “the super call.”

⚠ Exam Trap — missing no-arg parent constructor

If the parent defines a constructor with args and no no-arg constructor, the auto-inserted super() has nothing to call:

public class Mammal { public Mammal(int age) {} }
public class Seal extends Mammal {}              // DOES NOT COMPILE
public class Elephant extends Mammal {
    public Elephant() {}                          // DOES NOT COMPILE — implicit super() has no target
}

Fix: add an explicit super(6); in the child constructor. (A subclass may have a no-arg constructor even when its parent doesn’t — as long as it calls an existing parent constructor.)

SECTION 04

Initializing Objects

Order of class initialization

Initialize Class X (happens at most ONCE)

  1. Initialize the superclass of X first.
  2. Process all static variable declarations in source order.
  3. Process all static initializers in source order.

A class loads before it is first used; the class holding main() loads before main() runs. If main() is inside Hippo (which extends Animal), the program prints ABC exactly once — A+B from class init, then C from main(), even though three instances are created. If Hippo is instead used from another class, you’d likely see CAB (loaded only when needed).

final fields

✓ Key Rule

By the time a constructor completes, every final instance variable must be assigned a value exactly once — at its declaration, in an instance initializer, or in the constructor. Default values apply only to non-final fields.

public class MouseHouse {
    private final int volume;
    private final String type;
    { this.volume = 10; }                // instance initializer sets volume
    public MouseHouse(String type) { this.type = type; }   // OK
    public MouseHouse() {                  // DOES NOT COMPILE
        this.volume = 2;                   // volume already set; AND type never set
    }
}

Each constructor is checked individually. The no-arg version fails twice: it never sets type, and it re-assigns the already-final volume. Delegating with this(null) fixes it — null is a valid one-time assignment for a final reference.

Order of instance initialization

Initialize Instance of X

  1. Initialize Class X (if not already done).
  2. Initialize the superclass instance of X.
  3. Process instance variable declarations in source order.
  4. Process instance initializers in source order.
  5. Run the constructor (including any this()-chained constructors).

Short form: declarations → initializers → constructor, superclass before subclass.

TRACE — ZooTickets → prints 0-10-BestZoo-z-

  • No superclass declared (so Object). Static first: COUNT=0, static init prints 0-, second static init does COUNT+=10 & prints 10-.
  • Instance next: name="BestZoo", instance init prints BestZoo-.
  • Constructor: prints z-. ⇒ 0-10-BestZoo-z-

TRACE — Primate / Ape / Chimpanzee → prints Primate-Ape1-Chimpanzee-

  • new Chimpanzee() → super(2) calls Ape(int).
  • Ape(int) has an inserted super() → Primate() prints Primate-.
  • Back in Ape(int): prints Ape1-. Then Chimpanzee(): prints Chimpanzee-.
  • Only ONE of the two Ape constructors runs. ⇒ Primate-Ape1-Chimpanzee-
⚠ Exam Trap — the hard one (Okapi)

Output is AFBECHG then BECHG. Walk it slowly:

  • Class init: superclass GiraffeFamily static prints A; Okapi static prints F.
  • new Okapi(1): super instance → GiraffeFamily instance init B; constructor GiraffeFamily(String) calls this(1) → GiraffeFamily(int) prints E, then back prints C ⇒ BEC.
  • Then Okapi instance init H, constructor G ⇒ BECHG. Combined first line: AFBECHG.
  • Second new Okapi(2): class already loaded, so just BECHG. D never prints — the no-arg GiraffeFamily() is never called.
SECTION 05

Inheriting Members

Overriding a method

Overriding = a subclass declares a new implementation for an inherited method with the same signature (name + parameters) and a compatible return type. Signature excludes return type, access modifiers, specifiers, and exceptions.

public class Kangaroo extends Marsupial {
    public double getAverageWeight() {
        return super.getAverageWeight() + 20;   // Marsupial=50.0, Kangaroo=70.0
    }
}
// Without super, calling getAverageWeight() inside itself => StackOverflowError at runtime

The 4 rules for overriding

  1. Same signature as the parent method.
  2. At least as accessible as the parent method (not more restrictive).
  3. May not declare a new or broader checked exception.
  4. Return type must be the same or a subtype (covariant).
Rule violatedExampleResult
#2 Accesspublic parent → private child getNumberOfHumps()No compile
#3 Exception (new)hide() adds throws FileNotFoundException (parent throws none)No compile
#3 Exception (broader)exitShell() throws IOException (parent throws FileNotFoundException)No compile
#3 Narrower exceptionsleep() throws FileNotFoundException (parent throws IOException)OK
#4 Covariant OKgetName(): String child overrides CharSequence parentOK
#4 Covariant failgetColor(): CharSequence child overrides String parentNo compile
★ Remember

Covariance test: can you assign a B to an A reference without a cast? If yes, B is covariant with A. String is a subtype of CharSequence, so String→CharSequence works but not the reverse. If a return type is void, the override must also be void. Overrides may freely add new unchecked exceptions.

Redeclaring private methods

You can’t override a private method (it isn’t inherited). A child can redeclare one with any signature/return type — it’s simply a new, independent method, no override rules apply.

public class Beetle { private String getSize() { return "Undefined"; } }
public class RhinocerosBeetle extends Beetle {
    private int getSize() { return 5; }   // compiles — not an override (parent's is private)
}

Hiding static methods

✓ Key Rule

Static methods can’t be overridden — they’re hidden. All 4 override rules apply, plus a 5th: the child method must be static iff the parent method is static. Mismatch (one static, one not) ⇒ does not compile.

public class Bear {
    public static void sneeze() {}
    public void hibernate() {}
    public static void laugh() {}
}
public class SunBear extends Bear {
    public void sneeze() {}            // DOES NOT COMPILE — parent static, child instance
    public static void hibernate() {}  // DOES NOT COMPILE — parent instance, child static
    protected static void laugh() {}   // DOES NOT COMPILE — more restrictive (rule #2)
}

Hiding variables

Java never overrides variables — it only hides them. A same-named child variable creates two distinct copies; which one you see depends on the reference type.

class Carnivore { protected boolean hasFur = false; }
public class Meerkat extends Carnivore {
    protected boolean hasFur = true;
    public static void main(String[] args) {
        Meerkat m = new Meerkat();
        Carnivore c = m;
        System.out.println(m.hasFur);   // true  (Meerkat reference)
        System.out.println(c.hasFur);   // false (Carnivore reference)
    }
}
★ Remember

Overriding a method replaces it for all references (except super). Hiding a method/variable applies only when a child reference type is used.

final methods

final methods cannot be overridden — nor can a final static method be hidden. Whether the child also writes final is irrelevant. (If the parent method were private, the child version would be a legal redeclaration.)

SECTION 06

Creating Abstract Classes

An abstract class (marked abstract) cannot be instantiated directly and may contain abstract methods — methods declared with abstract and no body, which force subclasses to implement them.

public abstract class Canine {
    public abstract String getSound();          // no body, ends in ;
    public void bark() { System.out.println(getSound()); }
}
// Canine w = new Fox(); w.bark();  => Squeak!

Abstract rules

  • Only instance methods can be abstract — not variables, constructors, or static methods.
  • An abstract class may have zero or more abstract methods; a non-abstract class may have none.
  • The first concrete subclass must implement all inherited abstract methods.
  • Overriding an abstract method follows the normal override rules.
  • An abstract class is initialized via constructors only when a concrete subclass is instantiated.
DeclarationWhy it fails
var a = new Alligator(); (Alligator abstract)Can’t instantiate an abstract class
FennecFox.getSound() returns intNot covariant with String
ArcticFox extends Canine {}Doesn’t implement getSound()
Direwolf (concrete) declares abstract rest();Abstract method in a non-abstract class
Jackal declares abstract String name;Variables can’t be abstract
public class Egret { abstract void peck(); }Abstract method outside an abstract class
✓ Key Rule

An abstract class need not contain any abstract methods (e.g. abstract class Llama { void chew(){} } compiles). But it still can’t be instantiated.

Spotting invalid abstract declarations

public abstract class Turtle {
    public abstract long eat()            // DOES NOT COMPILE — missing semicolon
    public abstract void swim() {};       // DOES NOT COMPILE — abstract + body
    public abstract int getAge() { return 10; }  // DOES NOT COMPILE — abstract + body
    public abstract void sleep;          // DOES NOT COMPILE — missing ()
    public void goInShell();              // DOES NOT COMPILE — not abstract, needs a body
}

The abstract modifier may sit before or after the access modifier (abstract public class Tiger is fine), but not after the class keyword (public class abstract Bear fails) or after a method’s return type.

Illegal modifier combinations

CombinationAllowed?Why
abstract + finalNoConflict: one demands extension, the other forbids it
abstract + privateNoA private method isn’t inherited, so it can’t be implemented
abstract + staticNoStatic methods can’t be overridden, so never implemented
final + privateYesLegal but redundant
⚠ Exam Trap — first concrete subclass

Abstract methods may be passed down through several abstract classes. The first concrete subclass must implement everything still abstract. If an intermediate abstract class implements one method, the concrete class only owes the rest. Turn an intermediate class concrete and it suddenly owes all remaining abstract methods.

SECTION 07

Creating Immutable Objects

An immutable object cannot change state after creation — great for security and for sharing safely across threads.

Common immutability strategy

  1. Mark the class final or make all constructors private.
  2. Mark all instance variables private and final.
  3. Don’t define any setter methods.
  4. Don’t allow referenced mutable objects to be modified.
  5. Use a constructor to set all fields, making a copy if needed.
⚠ Exam Trap — leaking a mutable reference

Even with rules 1–3 followed, returning the internal list directly breaks immutability:

public List<String> getFavoriteFoods() { return favoriteFoods; }  // caller can clear()/add()!

Safer alternatives: delegate methods (getFavoriteFoodsItem(int), getFavoriteFoodsCount()) or a copy-on-read getter: return new ArrayList<>(this.favoriteFoods);.

Defensive copy (rule 5)

If a caller supplies the data, storing their reference lets them mutate it later. Copy it in the constructor:

public Animal(List<String> favoriteFoods) {
    if (favoriteFoods == null || favoriteFoods.size() == 0)
        throw new RuntimeException("favoriteFoods is required");
    this.favoriteFoods = new ArrayList<String>(favoriteFoods);  // defensive copy
}
★ Remember

Without the defensive copy, a caller keeping their own reference can call clear()/add() afterward and the “immutable” object changes — the classic exam trap.

RECAP

Short Recap

Inheritance

  • extends = subclass; parent can’t be final. public/protected always inherited, package only same-package, private never.
  • All classes inherit java.lang.Object; primitives don’t.

Creating classes

  • One public top-level class per file; top-level can’t be protected/private.
  • this = current+inherited; super = inherited only; both unavailable in static context (only this).

Constructors

  • Name matches class, no return type, no var params. Default no-arg only when zero constructors declared.
  • First line is this() or super() (else compiler adds super()); must be first statement; no cycles.

Initialization order

  • Class: superclass → static vars → static initializers (once). Instance: superclass instance → instance vars → instance initializers → constructor.
  • Every final field assigned exactly once by constructor’s end.

Overriding / hiding

  • Override: same signature, ≥ access, no new/broader checked exceptions, covariant return.
  • Static → hidden (must match static); variables → hidden (reference type decides); final can’t be overridden/hidden.

Abstract classes

  • Can’t instantiate; abstract methods have no body and end in ;. First concrete subclass implements all. No abstract+final/private/static.

Immutable objects

  • final class or private constructors, private final fields, no setters, no mutable leaks, defensive copies in/out.

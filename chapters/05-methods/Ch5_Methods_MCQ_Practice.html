Q1Designing

Which method declaration does not compile?

  • A public final void run() {}
  • B final public void run() {}
  • C public void final run() {}
  • D void run() {}
Show Answer
C · Does not compile

Optional specifiers (like final) must come before the return type. In C, final appears after void. B is legal — a specifier may precede the access modifier; D uses package access, which is fine.

Q2Designing

How many of these methods compile?

public String a() {}
public void b() { return; }
public String c() { return ""; }
String d(int x) { if(1 < 2) return "hi"; }
  • A 1
  • B 2
  • C 3
  • D 4
Show Answer
B · 2

Only b() and c() compile. a() declares String but returns nothing. d() returns only inside the if — the compiler requires a guaranteed return on every path and won't pre-evaluate 1<2.

Q3Designing

Which pair of methods has the same signature (so they can't coexist in one class)?

  • A void fly(int a, int b) & void fly(int b, int a)
  • B void fly(int a) & void fly(long a)
  • C void fly(int a) & void fly(int... a)
  • D void fly(String a) & void fly(int a)
Show Answer
A · same signature

A signature is the name plus parameter types and order; names are irrelevant. A has identical types (int, int), so it's a duplicate. B/D differ by type, and C differs too (int vs int[]).

Q4Designing

Which method declaration is legal?

  • A public void 2start() {}
  • B public void _() {}
  • C public void my$Method() {}
  • D public int go() {}
Show Answer
C · legal

Identifiers may contain $ and underscores, so my$Method is fine. A starts with a digit; a single underscore _ is reserved (B); D declares int but never returns a value.

Q5Designing

Why does this fail to compile?

public void run(int a; int b) {}
  • A It compiles fine
  • B Parameters must be separated by commas, not semicolons
  • C void methods can't take parameters
  • D The two parameters must have different types
Show Answer
B · comma, not semicolon

Parameters in a list are comma-separated. A semicolon is a syntax error. void methods can take parameters, and parameters may share a type.

Q6Variables

Choose all that apply. Which variables are effectively final?

String name = "Hippo";
var size = 10;
boolean wet;
if(size > 100) size++;
name.substring(0);
wet = true;
  • A name
  • B size
  • C wet
Show Answer
A, C · name and wet

name is assigned once (substring returns a new String that's discarded — String is immutable). wet is assigned once. size may be reassigned by size++, so it's not effectively final.

Q7Variables

What is the result?

final int rest;
if(isWeekend) rest = 5;
System.out.print(rest);
  • A Compiles and prints 5
  • B Compiles and prints 0
  • C Does not compile — rest may be unassigned
  • D Does not compile — a local can't be final
Show Answer
C · Does not compile

If isWeekend is false, rest is never assigned, and a final variable has no default. The compiler rejects use of a possibly-unassigned variable. (Locals may absolutely be final.)

Q8Variables

What happens with these two lines?

final int[] nums = new int[3];
nums[0] = 1;        // line A
nums = new int[5];  // line B
  • A Both lines compile
  • B Line A compiles; line B does not
  • C Line B compiles; line A does not
  • D Neither compiles
Show Answer
B · contents OK, reassign fails

final locks the reference, not the contents. Writing nums[0] mutates the array (fine); reassigning nums to a new array repoints the reference (not allowed).

Q9Variables

What is the result?

public class Bear {
  final int age;
  public Bear() { }
}
  • A Compiles; age defaults to 0
  • B Compiles; age is null
  • C Does not compile — final field is never assigned
  • D Throws an exception at runtime
Show Answer
C · Does not compile

A final instance field gets no default and must be assigned at declaration, in an instance initializer, or in every constructor. The empty constructor leaves age unassigned.

Q10Varargs

Which method declaration does not compile?

  • A void m(int... a)
  • B void m(String s, int... a)
  • C void m(int... a, String s)
  • D void m(int[] a, String s)
Show Answer
C · varargs not last

A varargs parameter must be the last one. D is fine — a plain array parameter has no positional restriction.

Q11Varargs

What is printed?

static void go(int x, int... nums) {
  System.out.print(nums.length);
}
go(1); go(1,2); go(1,2,3);
  • A 012
  • B 123
  • C 122
  • D 111
Show Answer
A · 012

The first argument fills x; the rest form the array. So the lengths are 0, 1, and 2. An absent varargs becomes a length-0 array, not null.

Q12Varargs

What is the result?

static void go(int x, int... nums) {
  System.out.print(nums.length);
}
go(1, null);
  • A Prints 0
  • B Prints 1
  • C Does not compile
  • D Throws NullPointerException
Show Answer
D · NullPointerException

null isn't an int, so Java passes it as the array reference itself (which is null). It compiles; then nums.length dereferences null at runtime.

Q13Varargs

Given void m(int... a), which call(s) make a a length-0 array?

  • A m()
  • B m(null)
  • C m(new int[0])
  • D Both A and C
Show Answer
D · both A and C

Passing no values (m()) or an explicit empty array (m(new int[0])) both yield a length-0 array. m(null) makes a null, not empty.

Q14Access

Bird is in package a with protected void fly(). Which lines compile inside Swan (package b)?

package b;
public class Swan extends Bird {
  void test() {
    Bird bird = new Bird();
    bird.fly();      // line X
    Swan s = new Swan();
    s.fly();         // line Y
  }
}
  • A Both X and Y
  • B Only Y
  • C Only X
  • D Neither
Show Answer
B · only Y

From a subclass in another package, protected access through a variable requires the reference type to be the subclass (or lower). s is a Swan (OK); bird is a Bird parent reference (denied).

Q15Access

Choose all that apply. A method with package (default) access can be called from…

  • A the same class
  • B a different class in the same package
  • C a subclass in a different package
  • D an unrelated class in a different package
Show Answer
A, B · same class & same package

Package access reaches the same class and any class in the same package — and stops there. Crossing package boundaries (C and D) needs protected or public.

Q16Access

A and B are both in package zoo. What is the result?

public class A { private int x = 5; }

public class B {
  void test() {
    A a = new A();
    System.out.print(a.x);
  }
}
  • A Prints 5
  • B Does not compile — x is private
  • C Prints 0
  • D Throws an exception
Show Answer
B · private is class-only

private restricts access to the same class — being in the same package doesn't help. B cannot read A.x.

Q17Static

What is printed?

public class Snake {
  static int hiss = 2;
  public static void main(String[] a) {
    Snake s = new Snake();
    s = null;
    System.out.println(s.hiss);
  }
}
  • A 2
  • B 0
  • C NullPointerException
  • D Does not compile
Show Answer
A · 2 (no NPE)

A static member accessed through a reference uses the reference's type, not the object — so the null reference is never dereferenced. It reads Snake.hiss, which is 2.

Q18Static

What is printed?

public class Counter {
  static int count;
  public Counter() { count++; }
  public static void main(String[] a) {
    new Counter(); new Counter(); new Counter();
    System.out.print(count);
  }
}
  • A 0
  • B 1
  • C 3
  • D Does not compile
Show Answer
C · 3

There's one shared count for the whole class. Each constructor call increments the same variable, so after three objects it's 3.

Q19Static

What is the result?

public class Zoo {
  private String name = "Leo";
  public static void main(String[] a) {
    System.out.print(name);
  }
}
  • A Prints Leo
  • B Prints null
  • C Does not compile — static main can't use instance variable name
  • D Throws an exception
Show Answer
C · Does not compile

A static method has no implicit object, so it can't reference the instance variable name without a reference. Either make name static or access it via new Zoo().name.

Q20Static

Which line(s) cause a compile error?

1: static final int a;
2: static final int b = 2;
3: static { a = 1; b = 3; }
  • A Only b = 3 on line 3
  • B Lines 2 and 3
  • C Lines 1 and 3
  • D Nothing — it all compiles
Show Answer
A · only b = 3

a is final-but-unassigned, so the static initializer setting a = 1 is its one valid assignment. b was already assigned on line 2, so b = 3 is a forbidden second assignment.

Q21Static

Which import statement is valid?

  • A import static java.util.Arrays;
  • B static import java.util.Arrays.asList;
  • C import static java.util.Arrays.asList;
  • D import java.util.Arrays.* static;
Show Answer
C · valid

The keyword order is import static, and it must target a member, not a class. A imports a class (invalid for static import); B has the keywords reversed; D is malformed.

Q22Pass-by-value

What is printed?

public static void main(String[] a) {
  int x = 5;
  StringBuilder sb = new StringBuilder("Hi");
  modify(x, sb);
  System.out.println(x + sb);
}
static void modify(int x, StringBuilder sb) {
  x = 10;
  sb.append("There");
}
  • A 5HiThere
  • B 10HiThere
  • C 5Hi
  • D 10Hi
Show Answer
A · 5HiThere

Java is pass-by-value. Reassigning the int parameter doesn't touch the caller's x (stays 5). But append mutates the shared StringBuilder, so it becomes HiThere.

Q23Pass-by-value

What is printed?

String s = "abc";
s.concat("d");
s.toUpperCase();
System.out.println(s);
  • A abc
  • B abcd
  • C ABCD
  • D ABC
Show Answer
A · abc

String is immutable: concat and toUpperCase each return a new String that is discarded because the return value isn't assigned. The original s is unchanged.

Q24Autoboxing

Which assignment does not compile?

  • A long x = Integer.valueOf(5);
  • B Long y = 5L;
  • C Long z = 5;
  • D Integer w = 5;
Show Answer
C · widen + box at once

5 is an int; assigning it to Long needs widening (int→long) and boxing in one step, which Java won't do. A unboxes then widens (allowed); B and D are plain autoboxing.

Q25Autoboxing

What is the result?

Integer count = null;
int x = count;
System.out.println(x);
  • A Prints 0
  • B Prints null
  • C Does not compile
  • D Throws NullPointerException
Show Answer
D · NullPointerException

Assigning a null wrapper to a primitive requires unboxing null, which throws NullPointerException at runtime. It compiles fine — that's the trap.

Q26Overloading

What is printed?

static void p(int i)    { System.out.print("int"); }
static void p(long l)   { System.out.print("long"); }
static void p(Object o) { System.out.print("obj"); }
public static void main(String[] a) { p(5); }
  • A int
  • B long
  • C obj
  • D Does not compile
Show Answer
A · int

5 is an int literal, so the exact-match p(int) wins. Widening to long or boxing to Object are only considered when no exact match exists.

Q27Overloading

What is printed?

static void p(Integer i) { System.out.print("Integer"); }
static void p(long l)    { System.out.print("long"); }
public static void main(String[] a) {
  int x = 5;
  p(x);
}
  • A Integer
  • B long
  • C Does not compile
  • D Ambiguous — won't compile
Show Answer
B · long

For an int argument, widening to long is preferred over autoboxing to Integer. Boxing is a later step in the resolution order, so p(long) wins.

Q28Overloading

Which pair does not compile together?

  • A void m(int a) & void m(int a, int b)
  • B void m(int a) & long m(int a)
  • C void m(int a) & void m(double a)
  • D void m(String a) & void m(StringBuilder a)
Show Answer
B · differ only by return type

B has the same name and the same parameter list, differing only in return type — that's not overloading, it's a duplicate method. The others have distinct parameter lists.

Q29Overloading

What is printed?

static void g(int... a)      { System.out.print("var"); }
static void g(int a, int b) { System.out.print("two"); }
public static void main(String[] x) { g(1, 2); }
  • A var
  • B two
  • C Does not compile
  • D Ambiguous — won't compile
Show Answer
B · two

An exact fixed-arity match (g(int, int)) always beats a varargs method. Varargs is the last resort in overload resolution.

Quick Revision — core traps

  • Declarations: modifiers/specifiers go before the return type; default isn't an access modifier; signature = name + parameter types & order (names don't count).
  • final: locks the reference, not the contents; a final field/local has no default and must be assigned before use. Effectively final = never reassigned.
  • Varargs: at most one, must be last; no args → length-0 array; passing null compiles, then throws NPE.
  • Access: private is class-only even in the same package; package access has no keyword; protected through a variable checks the declared reference type.
  • Static: belongs to the class; access via a null reference gives no NPE; a static member can't reach instance members without a reference; final static may be assigned once.
  • Pass-by-value: reassigning a parameter never affects the caller; mutating a shared object does; ignored return values change nothing (immutability of String).
  • Autoboxing: widen or box, never both at once (Long x = 8; fails); unboxing null → NPE.
  • Overloading: resolution order is exact → widening → autoboxing → varargs; differing only by return type is a duplicate method.

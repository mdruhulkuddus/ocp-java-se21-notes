SECTION 01

Designing Methods

Before anything else, you need to recognize a valid method declaration on sight — the exam constantly shows you slightly-broken ones. A declaration has parts that come in a fixed order once you reach the return type. Here is the full anatomy:

public final void nap(int minutes) throws InterruptedException { }
//  ^access  ^specifier  ^return  ^name  ^params           ^exception list   ^body
PartExampleRequired?
Access modifierpublicNo
Optional specifierfinalNo
Return typevoidYes
Method namenapYes
Parameter list(int minutes)Yes (may be empty)
Exception listthrows InterruptedExceptionNo
Method body{ }Yes (except abstract)

Access modifiers & optional specifiers

The four access levels are public, protected, private, and package (no keyword) — full details in Section 4. The optional specifiers are extra keywords like static, abstract, final, default, synchronized, native, and strictfp. Only the first few are on the exam.

public class ParkTrip {
  public void skip1() {}
  default void skip2() {} // DOES NOT COMPILE - default is for interfaces, not classes
  void public skip3() {} // DOES NOT COMPILE - access modifier after the return type
  void skip4() {}          // OK - package access (no keyword)
}
Key Rule — ordering

Access modifier and optional specifiers may appear in any order relative to each other, and a specifier may even come before the access modifier. But all of them must come before the return type. Nothing extra may appear after the return type except the name.

public static final void bike3() {}  // OK
public final static void bike4() {}  // OK - specifier order is free
final public void bike7() {}        // OK - specifier before access modifier (legal!)
public modifier void bike5() {}     // DOES NOT COMPILE - 'modifier' isn't a real specifier
public void final bike6() {}        // DOES NOT COMPILE - specifier after the return type
Exam Trap — default and abstract

default is a real keyword (used in switch and interface methods) but it is never a method access modifier — package access has no keyword. Also, abstract and final can't be combined (an abstract method must be overridden; a final one can't be), and an abstract method has no body.

Return type & return statements

Key Rule

Every method must declare a return type — use void if nothing comes back. A non-void method must have a reachable return that returns a value assignable to that type.

public String hike4() {}            // DOES NOT COMPILE - missing return
public hike5() {}                   // DOES NOT COMPILE - missing return type
public String int hike6() {}        // DOES NOT COMPILE - two return types
String hike7(int a) {              // DOES NOT COMPILE - return only inside the if
  if (1 < 2) return "orange";
}

int getHeight2() { int t = 9L; return t; }  // DOES NOT COMPILE - long does not fit int
int getHeight3() { long t = 9L; return t; } // DOES NOT COMPILE - returning long as int
Remember

The compiler will not pre-evaluate if(1<2) as "always true"; it still demands a guaranteed return on every path. Code after a guaranteed return is a warning (dead code), not an error.

Method name

A method name follows the same rules as any identifier: it must start with a letter, $, or _; subsequent characters may include digits; it can't be a reserved word; and a single underscore _ is illegal on its own.

public void jog1() {}    // OK
public void 2jog() {}    // DOES NOT COMPILE - starts with a digit
public void Jog_$() {}   // OK - ugly but legal
public void _() {}       // DOES NOT COMPILE - a single underscore is reserved

Parameter list, exceptions & body

The parameter list is required but may be empty (). Multiple parameters are separated by commas. The optional exception list (throws ...) comes after the parameters — covered in Chapter 11. The body is braces { }, required for every method except abstract ones.

public void run4(int a; int b) {} // DOES NOT COMPILE - use ',' not ';'

Method signature

The signature is the method name plus the parameter list. Crucially, only the parameter types and their order count — parameter names are not part of the signature. Two methods with the same signature can't coexist.

// DOES NOT COMPILE - same signature (names are irrelevant)
public void visitZoo(String name, int waitTime) {}
public void visitZoo(String attraction, int rainFall) {}

// OK - the type order differs, so the signatures differ
public void visitZoo(String name, int waitTime) {}
public void visitZoo(int rainFall, String attraction) {}
SECTION 02

Local & Instance Variables

The variables you use inside a method versus the ones that live on a class behave differently. The two things that set them apart are where they can be seen (scope) and how long they live (lifetime). Look at this Lion example:

public class Lion {
  int hunger = 4;                 // instance variable

  public int feedZooAnimals() {
    int snack = 10;             // local variable
    if(snack > 4) {
      long dinnerTime = snack++; // local variable (only in this if-block)
      hunger--;
    }
    return snack;
  }
}

Local variable

  • Declared inside a method/block
  • Only modifier allowed: final
  • No default value — must be assigned before use
  • Reference destroyed when the block ends

Instance variable

  • A member of the class, one per object
  • Modifiers: final, volatile, transient (+ access)
  • Gets a default value (0, null, false...)
  • If final, gets no default

The only modifier on a local variable: final

final int rest;
if(isWeekend) rest = 5; else rest = 20;
System.out.print(rest);            // OK - assigned on both paths

final var giraffe = new Animal();
giraffe = new Animal();           // DOES NOT COMPILE - reassigning a final

Note that final int rest; needs no value at declaration. The rule is just: assign it exactly once before use. If a path leaves it unassigned, it won't compile.

Exam Trap — final reference vs contents

final locks the reference, not the object's internals:

final int[] friends = new int[5];
friends[2] = 2;     // OK - mutate array contents
friends = null;      // DOES NOT COMPILE - repointing the variable

Effectively final

Remember — the quick test

A local variable is effectively final if it's never reassigned after its first value. Test: add final; if it still compiles, it was effectively final.

12: String name = "Harry the Hippo";  // effectively final
13: var size = 10;
15: if(size > 100) size++;          // size CAN change -> NOT effectively final
16: name.substring(0);              // returns a new String, discarded; name unchanged
17: boolean wet = true;             // effectively final
Remember — why it matters

Method/constructor parameters are really pre-initialized locals. The final / effectively-final rule becomes important with lambdas and local classes (Ch. 7–8), which may only capture final or effectively-final locals.

Final instance variables

A final instance variable must be assigned exactly once, in one of three spots — at declaration, in an instance initializer, or in a constructor.

public class PolarBear {
  final int age = 10;     // (1) at declaration
  final int fishEaten;     // (2) in an instance initializer
  final String name;       // (3) in the constructor
  { fishEaten = 10; }
  public PolarBear() { name = "Robert"; }
}
Exam Trap — final has no default

Ordinary fields get a default (int→0, object→null). The compiler gives a final field no default, so an unassigned final instance/static variable won't compile.

SECTION 03

Working with Varargs

Varargs (variable arguments) is a parameter that lets you pass any number of values — zero or more. Inside the method it behaves like an array. You write it with three dots: int... steps.

Two rules to remember

  1. A method may have at most one varargs parameter.
  2. The varargs parameter must be the last one in the list.
public void walk1(int... steps) {}            // OK
public void walk2(int start, int... steps) {} // OK - varargs is last
public void walk3(int... steps, int start) {} // DOES NOT COMPILE - varargs not last
public void walk4(int... a, int... b) {}      // DOES NOT COMPILE - two varargs

Two ways to call

Pass a ready-made array, or pass individual values (Java builds the array). Pass nothing and you get a length-0 array, never null.

public static void walkDog(int start, int... steps) {
  System.out.println(steps.length);
}
walkDog(1);                  // 0
walkDog(1, 2);               // 1
walkDog(1, 2, 3);            // 2
walkDog(1, new int[]{4,5}); // 2 - array passed directly
  • The first argument fills start; whatever remains becomes the steps array.
  • You can read steps.length and index into it (steps[1]) just like an array.
Exam Trap — passing null 
walkDog(1, null); // compiles, then NullPointerException

null isn't an int, so Java treats it as the array reference itself (which is null). It compiles; then steps.length throws NullPointerException at runtime.

SECTION 04

Applying Access Modifiers

An access modifier is like a security guard — it decides which classes may touch a member. Four levels, strictest to most open:

The four access levels

  • private — only inside the same class.
  • package (no keyword) — same class + other classes in the same package.
  • protected — package access + subclasses in a different package.
  • public — from anywhere.
Accessing a member from…privatepackageprotectedpublic
the same classYesYesYesYes
another class, same packageNoYesYesYes
a subclass, different packageNoNoYesYes
unrelated class, different packageNoNoNoYes

The tricky protected gotcha

From a subclass in a different package, protected access depends on how you reach the member:

1: package pond.swan;          // different package from Bird
3: public class Swan extends Bird {
5:   floatInWater();           // OK - inheritance, no variable
10:  Swan s = new Swan();
11:  s.floatInWater();         // OK - reference type is Swan (a subclass)
15:  Bird b = new Bird();
16:  b.floatInWater();         // DOES NOT COMPILE - reference type is Bird
18: }
Exam Trap — two scenarios

No variable (inheritance): allowed. Through a variable: the reference type must be the same class or a subclass — a parent (Bird) reference is denied, even if the object is actually a Swan. It's the declared type that's checked, not the runtime object.

SECTION 05

Accessing Static Data

The static keyword makes a variable, method, or nested class belong to the class rather than any single instance. All objects share one copy. main() is static, which is why the JVM can call it without an object.

class Penguin {
  String name;                       // per object
  static String nameOfTallestPenguin; // shared by all
}
p1.nameOfTallestPenguin = "Lilly";
p2.nameOfTallestPenguin = "Willy";
System.out.println(p1.nameOfTallestPenguin); // Willy - shared, last value wins

Static access via a reference — the null trick

Exam Trap — no NullPointerException 
Snake s = new Snake();
System.out.println(s.hiss); // 2
s = null;
System.out.println(s.hiss); // 2 - still works, hiss is static

Accessing a static member through an instance uses the reference's type, not the object — so even a null reference works. Java never dereferences it.

And there is only ever one shared copy:

Snake.hiss = 4;
Snake a = new Snake(); a.hiss = 6;
Snake b = new Snake(); b.hiss = 5;
System.out.println(Snake.hiss); // 5  (4 -> 6 -> 5)

Class vs. instance membership

Key Rule

A static member can't reach an instance member without an object reference. An instance member can freely use static members.

public static void announceToEveryone() {
  addGorilla();   // OK - static method
  babyGorilla();  // DOES NOT COMPILE - static -> instance method
}
public static double average = total / count; // DOES NOT COMPILE - static uses instance var

Constants & static final

private static final int NUM_BUCKETS = 45;
NUM_BUCKETS = 5;  // DOES NOT COMPILE - reassigning a final

private static final String[] treats = new String[10];
treats[0] = "popcorn";  // OK - reference final, contents mutable

Static initializers

15: private static final int two;
16: private static final int three = 3;
17: private static final int four;  // DOES NOT COMPILE - never initialized
18: static {
20:   two = 2;     // OK - first assignment of a final
21:   three = 3;   // DOES NOT COMPILE - already assigned on line 16
22:   two = 4;     // DOES NOT COMPILE - second assignment of two
23: }
Remember

A final static may be assigned once in a static initializer (that counts as its assignment). Static initializers run when the class first loads, in declaration order.

Static imports

import static java.util.Arrays.asList; // OK
import static java.util.Arrays;        // DOES NOT COMPILE - members only, not a class
static import java.util.Arrays.*;      // DOES NOT COMPILE - it's "import static"
Arrays.asList("one"); // DOES NOT COMPILE - the Arrays class itself wasn't imported
Exam Trap

The order is always import static. A static import brings in members, never a class. Importing two same-named members from different classes is an error.

SECTION 06

Passing Data among Methods

Java is strictly pass-by-value: a method always receives a copy of the variable. This one idea explains every "what is printed?" puzzle in this section.

Primitives — the copy is obvious

public static void main(String[] a) {
  int num = 4;
  newNumber(num);
  System.out.print(num); // 4 - caller is untouched
}
public static void newNumber(int num) { num = 8; }

The method gets a copy of 4. Reassigning the parameter changes only the copy, so the caller's num stays 4.

Objects — the reference is copied too

With objects, the reference is passed by value. The copy points at the same object, so two behaviors differ sharply:

Reassigning the parameter

var name = "Webby";
speak(name);
// name still "Webby"

void speak(String s){
  s = "Georgette"; // only the copy
}

Mutating the shared object

var sb = new StringBuilder("Webby");
speak(sb);
// prints WebbyGeorgette

void speak(StringBuilder s){
  s.append("Georgette"); // same object
}
Key Rule

Reassigning a parameter never affects the caller. Calling a mutating method on the shared object does, because both references point to the same object.

Returning data — you must use the result

Exam Trap — ignored return values

A returned value changes nothing unless you assign it.

int tickets = 2;
String guests = "abc";
addTickets(tickets);            // return ignored -> tickets stays 2
guests = addGuests(guests);     // assigned -> guests = "abcd"
System.out.println(tickets + guests); // 2abcd

Autoboxing & unboxing

Autoboxing converts a primitive to its wrapper (int→Integer) automatically; unboxing goes the other way. The rule that traps everyone:

Key Rule — one conversion at a time

Java will widen a primitive or autobox it, but not both in a single step.

long ears = e;          // OK - unbox Integer->int, then widen int->long
Long badGorilla = 8;  // DOES NOT COMPILE - would widen AND box at once

Character elephant = null;
char bad = elephant;    // NullPointerException - unboxing null

c.jump(123L); // DOES NOT COMPILE - jump(int): long->int is narrowing
g.rest(8);    // DOES NOT COMPILE - rest(Long): int->Long needs widen + box

Integer[] winterHours = { 10.5, 17.0 }; // DOES NOT COMPILE - double is not Integer
Remember

Unboxing a null wrapper into a primitive always throws NullPointerException at runtime — it compiles fine, which is exactly what makes it a trap.

SECTION 07

Overloading Methods

Overloading means several methods with the same name in one class but different parameter lists. (Different from overriding — Chapter 6.) println() and StringBuilder's append() are heavily overloaded, which is why they accept almost any type.

Key Rule

Everything except the method name and parameter list may differ — return type, access modifier, specifier, exceptions. But to be an overload, the parameter list must differ.

public void fly(int n) {}
public int fly(int n) { return 1; } // DOES NOT COMPILE - only return type differs
public static void fly(int n) {}     // DOES NOT COMPILE - same signature
public void fly(int km) {}           // DOES NOT COMPILE - param name doesn't count

public void fly(int[] lengths) {}
public void fly(int... lengths) {}     // DOES NOT COMPILE - varargs = array signature

How Java picks the overload

Resolution order (most → least specific)

  1. Exact match by type
  2. Larger primitive type (widening)
  3. Autoboxed type
  4. Varargs

Autoboxing and varargs come last for backward compatibility.

// reference types:  fly(String) vs fly(Object)
p.fly("test"); // "string" - String matches directly
p.fly(56);     // "object" - int boxes to Integer, not a String

// primitives:  fly(int) vs fly(long)
p.fly(123);  // "int"  - exact
p.fly(123L); // "long" - exact

// glide(String), glide(String...), glide(String,String)
glide("a");          // 1 - single String, most specific
glide("a","b");      // 4 - two-String beats varargs
glide("a","b","c");  // 2 - only varargs matches => prints 142
Remember

When both fly(int) and fly(Integer) exist, fly(3) picks the primitive — Java avoids needless autoboxing. Arrays of primitives and wrappers (int[] vs Integer[]) are distinct, with no autoboxing between them.

SECTION 08

Short Recap

Designing Methods

  • Order after the return type is fixed: return type → name → params → exceptions → body. Modifiers/specifiers go before the return type, in any order.
  • default isn't an access modifier; package access has no keyword. Signature = name + parameter types & order (names don't count).

Local & Instance Variables

  • Local: only final, no default, must assign before use. Instance: gets a default — unless final, which gets none.
  • final locks the reference, not the contents. Effectively final = never reassigned (matters for lambdas).

Varargs

  • At most one, and last. Acts like an array inside. No args → length-0 array; null → compiles then NPE.

Access Modifiers

  • private → package → protected → public, widening reach. protected through a variable checks the declared reference type, not the object.

Static Data

  • Belongs to the class, one shared copy. Static via a null reference → no NPE. Static can't reach instance members without a reference. final static = constant; assign once.

Passing Data among Methods

  • Pass-by-value always. Reassigning a parameter never affects the caller; mutating a shared object does. Ignored return values change nothing.
  • One conversion at a time (widen OR box, not both). Unboxing null → NPE.

Overloading

  • Same name, different parameter list. Return type / static alone don't overload. Resolution: exact → widen → autobox → varargs; most-specific wins.

SECTION 01

Writing Lambdas

A lambda expression is a block of code that gets passed around — an unnamed method with parameters and a body, living inside an anonymous class. Lambdas enable deferred execution: code specified now but run later.

// Instead of a whole CheckIfHopper class, just pass the logic:
print(animals, a -> a.canHop());
print(animals, a -> a.canSwim());
print(animals, a -> !a.canSwim());

The syntax (and its optional parts)

These two lines do exactly the same thing. The arrow operator -> separates the parameter list from the body.

a -> a.canHop()                          // shortest form
(Animal a) -> { return a.canHop(); }     // most verbose form

What you can omit

  • Parentheses — only when there is a single parameter and its type is not stated.
  • Braces, return, and the semicolon — only for a single-expression body (no braces).
  • You don't have to use every declared parameter. And s -> {} is a valid lambda.

Valid (return boolean)

  • () -> true
  • x -> x.startsWith("test")
  • (String x) -> x.startsWith("test")
  • (x, y) -> { return x.startsWith("test"); }

Invalid

  • x, y -> ... — missing parentheses
  • x -> { x.startsWith("c"); } — missing return
  • x -> { return x.startsWith("g") } — missing semicolon
  • String x -> ... — type stated needs parentheses
Exam Trap — lambda + var

var invalid = (Animal a) -> a.canHop(); does not compile. A lambda gets its type from context, and var also needs context — together there is not enough information to know which functional interface to use.

SECTION 02

Functional Interfaces

A functional interface is an interface with exactly one abstract method — the single abstract method (SAM) rule. Lambdas and method references can only be assigned to functional interfaces.

@FunctionalInterface
public interface Sprint {
   public void sprint(int speed);
}
Key Rule — @FunctionalInterface

The annotation makes the compiler enforce the SAM rule (error if violated), but it is optional — an interface with one abstract method is functional whether or not it is annotated. @FunctionalInterface on interface Dance { void move(); void rest(); } does not compile (two abstract methods).

Which are functional interfaces?

Interface (given Sprint has sprint())Functional?Why
interface Dash extends Sprint {}Yesinherits the single abstract sprint()
interface Skip extends Sprint { void skip(); }Notwo abstract: sprint() + skip()
interface Sleep { private void snore(){} default int getZzz(){return 1;} }Nono abstract method at all
interface Climb { void reach(); default…; static…; private static…; }Yesonly one abstract: reach()

The Object-method exception

An abstract method whose signature matches a public method of Object — toString(), equals(Object), hashCode() — does not count toward the SAM total, because every implementing class already inherits it from Object.

public interface Soar { abstract String toString(); }   // NOT functional — toString doesn't count

public interface Dive {                // IS functional — dive() is the only SAM
   String toString();
   public boolean equals(Object o);
   public abstract int hashCode();
   public void dive();
}
Exam Trap — look-alike signatures

interface Hibernate { String toString(); boolean equals(Hibernate o); int hashCode(); void rest(); } is not functional. equals(Hibernate) ≠ equals(Object), so it counts — leaving two abstract methods (equals(Hibernate) and rest()). Also, an interface can't declare a method incompatible with Object (e.g. int toString() won't compile).

SECTION 03

Method References

A method reference (the :: operator) is a compact form of a lambda for deferred execution. A method reference and its equivalent lambda behave identically at runtime. There are four kinds.

TypeBefore ::ExampleEquivalent lambda
static methodClass nameMath::roundx -> Math.round(x)
instance method, bound objectvariable namestr::startsWiths -> str.startsWith(s)
instance method, unbound parameterClass nameString::isEmptys -> s.isEmpty()
constructorClass nameString::new() -> new String()
Key Rule — bound vs unbound

For an unbound reference like String::startsWith ↔ (s, p) -> s.startsWith(p), the first parameter is the instance the method is called on; the rest are arguments. It looks static but isn't.

Exam Trap — not every lambda becomes a reference

All method references convert to lambdas, but not vice versa. () -> str.startsWith("Zoo") cannot be a method reference — there's no way to also pass the "Zoo" argument. Both str::startsWith and str::startsWith("Zoo") fail to compile here.

Because overloaded methods exist (e.g. Math.round takes double or float), Java uses the functional-interface context to pick the right one. If it finds none or finds several, you get a compiler error (an ambiguous-type error).

SECTION 04

Built-in Functional Interfaces

The core interfaces live in java.util.function. <T> is the input, <U> a second input, <R> the return. Memorize this table — it is tested directly and underpins Chapters 9 and 10.

InterfaceMethodReturn# params
Supplier<T>get()T0

BiPredicate<T,U>test(T,U)boolean2 (T, U)

Supplier<LocalDate> s   = LocalDate::now;       // get()
Consumer<String> c      = System.out::println;   // accept("Annie")
Predicate<String> p     = String::isEmpty;       // test("")  -> true
Function<String,Integer> f = String::length;    // apply("cluck") -> 5
BiFunction<String,String,String> b = String::concat;
UnaryOperator<String> u = String::toUpperCase;   // apply("chirp") -> CHIRP
Exam Trap — Predicate vs Function

"Returns a Boolean and takes a String" is a Function<String,Boolean>, not a Predicate — a Predicate returns the boolean primitive. UnaryOperator/BinaryOperator force all type parameters to be the same type, so BinaryOperator<Integer> is preferred over the equivalent BiFunction<Integer,Integer,Integer>.

Function<List<String>> ex1 = x -> x.get(0);   // DOES NOT COMPILE — Function needs 2 type args
UnaryOperator<Long> ex2 = (Long l) -> 3.14;     // DOES NOT COMPILE — must return Long, not double

Convenience (default) methods

Combining functional interfaces

  • Predicate: and(), or(), negate() — e.g. egg.and(brown.negate()).
  • Consumer: andThen() — runs two consumers in sequence on the same input.
  • Function: andThen() (this, then arg) and compose() (arg first, then this).

andThen vs compose

  • c1.andThen(c2) then accept("Annie") → 1: Annie,2: Annie (same input to both).
  • after.compose(before) then apply(3) → before turns 3→4, after doubles 4→8.
SECTION 05

Primitive Functional Interfaces

To avoid boxing, there are primitive variants for double, int, and long — plus one for boolean. Two things change: the generics often disappear (the type is in the name), and the single abstract method is renamed when a primitive is returned.

Remember — BooleanSupplier

The only boolean functional interface is BooleanSupplier, whose method is getAsBoolean(). BooleanSupplier b = () -> true; then b.getAsBoolean().

SECTION 06

Variables in Lambdas

Variables show up in three places around a lambda: the parameter list, local variables declared in the body, and variables referenced from the body. Each is a favourite exam trap.

Parameter list — one format throughout

These three are interchangeable, and the inferred type of x is String:

Predicate<String> p = x -> true;
Predicate<String> p = (var x) -> true;
Predicate<String> p = (String x) -> true;
Key Rule — consistent format

Every parameter must use the same format: all untyped, all typed, or all var. You may add final or annotations to parameters: (final var x, @Deprecated var y) -> x.compareTo(y).

(var x, y) -> "Hello"                  // DOES NOT COMPILE — mix of var and none
(var x, Integer y) -> true             // DOES NOT COMPILE — mix of var and type
(String x, var y, Integer z) -> true   // DOES NOT COMPILE — three formats mixed
(Integer x, y) -> "goodbye"            // DOES NOT COMPILE — mix of type and none

Local variables in the body

(a, b) -> { int c = 0; return 5; }   // OK
(a, b) -> { int a = 0; return 5; }   // DOES NOT COMPILE — redeclares parameter a

Referencing surrounding variables

What a lambda body may access

  • Instance variables — always allowed.
  • Static variables — always allowed.
  • Local variables & method parameters — allowed only if final or effectively final.
  • Lambda parameters — always allowed.
Exam Trap — where the error lands

If a captured local/parameter is reassigned, the compiler error appears at the lambda's use of the variable, not at the reassignment line — even when the reassignment comes after the lambda. Assigning a value is only a problem from the lambda's point of view.

Short Recap

  • Lambda parentheses are optional only for a single untyped parameter; braces/return/semicolon are optional only for a single-expression body. var x = lambda; never compiles.
  • A functional interface has exactly one abstract method (SAM). @FunctionalInterface enforces but isn't required. Object-signature methods (toString, equals(Object), hashCode) don't count.
  • Four method-reference types: static (Math::round), bound instance (str::startsWith), unbound instance (String::isEmpty, first param is the instance), constructor (String::new).
  • Core interfaces: Supplier (0→T), Consumer/BiConsumer (→void), Predicate/BiPredicate (→boolean), Function/BiFunction (→R), UnaryOperator/BinaryOperator (same type).
  • Convenience: Predicate.and/or/negate, Consumer.andThen, Function.andThen/compose (compose runs the argument first).
  • Primitives drop generics and rename the method (getAsInt, applyAsDouble); BooleanSupplier is the only boolean one.
  • Lambda parameters share one format; a body can't redeclare a parameter; captured locals/params must be effectively final, with the error reported at the lambda.

SECTION 01

Implementing Interfaces

An interface is an abstract data type that declares a list of abstract methods any implementing class must provide. A class can extends only one class, but it may implements any number of interfaces, separated by commas. This is how Java gets multiple inheritance of type.

Declaring and using an interface

An interface is declared with the interface keyword. Interface variables are constants — implicitly public static final — and must be initialized with a constant value. Abstract methods are implicitly public.

public abstract interface WalksOnTwoLegs {}   // compiles; abstract is optional

final interface WalksOnEightLegs {}            // DOES NOT COMPILE — interface can't be final

public class Biped {
   public static void main(String[] args) {
      var e = new WalksOnTwoLegs();           // DOES NOT COMPILE — can't instantiate interface
   }
}
Exam Trap

An interface can no more be final than an abstract class can — marking it final implies nothing could ever implement it. And you can never write new SomeInterface().

A concrete class implementing an interface must override every inherited abstract method, declaring it explicitly public (the interface's implicit public is not inherited as a free pass — the override must restate it). Return types may be covariant.

public interface Climb {
   Number getSpeed(int age);
}
public class FieldMouse implements Climb {
   @Override
   public Float getSpeed(int age) { return 2.0f; }  // Float is covariant with Number; must be public
}

Extending interfaces & mixing keywords

An interface can extends multiple interfaces (unlike a class). The exam loves to swap extends / implements between classes and interfaces.

Allowed

  • class implements interface(s)
  • class extends one class
  • interface extends interface(s)

DOES NOT COMPILE

  • class extends interface
  • interface implements interface
  • interface extends a class
public interface CanRun {}
public class Cheetah extends CanRun {}      // DOES NOT COMPILE — class can't extend interface

public class Hyena {}
public interface HasFur extends Hyena {}     // DOES NOT COMPILE — interface can't extend class

Inheriting duplicate abstract methods

Two inherited abstract methods are fine if a single override can satisfy both (compatible signatures, covariant returns). Parameter names need not match — only types.

public interface Herbivore { int eatPlants(int plantsLeft); }
public interface Omnivore  { int eatPlants(int foodRemaining); }
public class Bear implements Herbivore, Omnivore {
   public int eatPlants(int plants) { return plants - 1; }  // satisfies both
}

// Incompatible — one returns void, the other int:
public interface Herbivore { void eatPlants(int p); }
public interface Omnivore  { int  eatPlants(int p); }
public class Tiger implements Herbivore, Omnivore {}  // DOES NOT COMPILE — impossible to override both

Implicit modifiers & conflicts

Implicit modifiers the compiler inserts

  • Interfaces are implicitly abstract.
  • Interface variables are implicitly public static final (constants).
  • Interface methods without a body are implicitly abstract.
  • Interface methods without private are implicitly public.

Because members are implicitly public, declaring them protected or private in a way that conflicts fails to compile. Interfaces do not support protected or package members.

public interface Dance {
   private int count = 4;       // DOES NOT COMPILE — conflicts with implicit public
   protected void step();        // DOES NOT COMPILE — conflicts with implicit public
}
Exam Trap — reducing visibility

An interface method is public. If an implementing class declares the override with weaker (package) access, it does not compile: class Georgette implements Poodle { void play() {} } fails because play() is public in Poodle. Overriding an abstract class's package-access method with package access is fine.

default methods

default method rules

  1. Declared only within an interface.
  2. Must use the default keyword and include a body.
  3. Implicitly public.
  4. Cannot be abstract, final, or static.
  5. May be overridden by an implementing class.
  6. If a class inherits two with the same signature, it must override.
public interface Walk { default int getSpeed() { return 5; } }
public interface Run  { default int getSpeed() { return 10; } }
public class Cat implements Walk, Run {}   // DOES NOT COMPILE — duplicate default, ambiguous

// Fix by overriding, and reach a "hidden" version with InterfaceName.super:
public class Cat implements Walk, Run {
   public int getSpeed() { return 1; }
   public int getWalkSpeed() { return Walk.super.getSpeed(); }  // not Walk.this
}
Key Rule

A default method belongs to an instance. You cannot call it on the interface: Dance.getRhythm() does not compile. Use the special InterfaceName.super.method() syntax to call an inherited (hidden) default.

static and private interface methods

static interface method rules

  1. Must use static and include a body.
  2. Implicitly public if no access modifier.
  3. Cannot be abstract or final.
  4. Not inherited — must be called with the interface name reference.
public interface Hop { static int getJumpHeight() { return 8; } }

public class Bunny implements Hop {
   public void printDetails() {
      System.out.println(getJumpHeight());        // DOES NOT COMPILE — needs Hop. prefix
      System.out.println(Hop.getJumpHeight());    // OK
   }
}

private and private static methods exist only to reduce duplication inside the interface. A private static method may be called by any method in the interface; a non-static private method may be called only by default and other non-static private methods. Neither is callable from an implementing class. Note: a static method cannot call a default/private instance method without an object reference.

SECTION 02

Working with Enums

An enum is a fixed set of constants, giving compile-time type safety. Values are commonly SNAKE_CASE. You compare them with == or equals() because each value is created once in the JVM.

public enum Season { WINTER, SPRING, SUMMER, FALL }

var s = Season.SUMMER;
System.out.println(Season.SUMMER);        // SUMMER (name via toString)
System.out.println(s == Season.SUMMER);   // true

Common methods

MethodReturnsNote
values()array of all valuesusable in a for-each loop
name()Stringthe declared name
ordinal()int0-based declaration order
valueOf(String)matching valuemust match exactly or throws IllegalArgumentException
 
Season.valueOf("SUMMER");   // SUMMER
Season.valueOf("summer");   // IllegalArgumentException at runtime
Exam Trap

You cannot compare an enum to an int or use its ordinal where a value is expected:

if (Season.SUMMER == 2) {} → DOES NOT COMPILE. Inside a switch, case 0 -> for an enum → DOES NOT COMPILE.

Enums can't be extended or marked final, but they can implement an interface.

public enum ExtendedSeason extends Season {}  // DOES NOT COMPILE — can't extend an enum

Enums in switch

Enum switches do not require a default branch if every value is handled (exhaustive). In Java 21 the enum name in the case label is optional.

String getWeather(Season value) {
   return switch (value) {
      case SUMMER        -> "Too hot";
      case Season.WINTER -> "Too cold";   // name optional but allowed
      case SPRING, FALL  -> "Just right";
   };  // no default needed — all values covered
}

Complex enums: fields, constructors, methods

Key Rule

For any enum with more than a value list, the list of values comes first and ends with a semicolon. Enum constructors are implicitly private; marking one public/protected does not compile. Each constructor runs once, the first time the enum is used.

enum SeasonWithVisitors implements Visitors {
   WINTER("Low"), SPRING("Medium"), SUMMER("High"), FALL("Medium");  // ; required
   private final String visitors;
   public static final String DESCRIPTION = "Weather enum";
   private SeasonWithVisitors(String visitors) {     // implicitly private
      System.out.print("constructing,");
      this.visitors = visitors;
   }
   @Override public void printVisitors() { System.out.println(visitors); }
}

CONSTRUCTOR RUNS ONCE PER VALUE

  • Calling SeasonWithVisitors.SUMMER twice prints begin,constructing,constructing,constructing,constructing,middle,end
  • All four values are constructed on first use; the second reference constructs nothing.

Value-specific method bodies

If the enum declares an abstract method, every value must implement it. Or provide a default body and override only special cases.

public enum SeasonWithTimes {
   WINTER { public String getHours() { return "10am-3pm"; } },
   SUMMER { public String getHours() { return "9am-7pm"; } },
   SPRING, FALL;
   public String getHours() { return "9am-5pm"; }   // default for the rest
}
SECTION 03

Sealing Classes

A sealed class restricts which classes may extend it. Three keywords matter: sealed (limits extension to named types), permits (lists those types), and non-sealed (reopens a sealed parent to unknown subclasses).

Sealed class rules

  • Declared with sealed and (optionally) permits.
  • The sealed class and its direct subclasses must be in the same package (or named module).
  • Every direct subclass must be exactly one of final, sealed, or non-sealed.
  • permits is optional if subclasses are in the same file or nested inside the sealed class.
  • Interfaces can be sealed too; extending interfaces may only be sealed or non-sealed (never final).
public class sealed Frog permits GlassFrog {}   // DOES NOT COMPILE — modifier order (sealed before class)

public abstract sealed class Mammal permits Wolf {}
public final class Wolf extends Mammal {}
public final class Tiger extends Mammal {}     // DOES NOT COMPILE — Tiger not in permits
Exam Trap — the subclass modifier

The most-tested rule: a direct subclass of a sealed class must declare final, sealed, or non-sealed. A plain class Sub extends SealedParent {} with none of these does not compile.

final subclass

  • Fixed — cannot be extended further.
  • final class Gazelle extends Antelope {}
  • class DamaGazelle extends Gazelle {} → DNC

sealed subclass

  • Must define its own permits list.
  • Indirect subclasses needn't appear in the top parent.
  • The full set of types is still fixed at compile time.
abstract sealed class Mammal permits Feline {}
non-sealed class Feline extends Mammal {}
class Tiger extends Feline {}            // OK — Feline is non-sealed
class BengalTiger extends Tiger {}       // OK — Tiger not final

Omitting permits & nested subclasses

The permits clause may be omitted when subclasses live in the same file as the sealed class, or are nested inside it. If you do name a nested subclass, you must qualify it with the enclosing namespace.

public sealed class Snake permits Cobra {
   final class Cobra extends Snake {}     // DOES NOT COMPILE — needs Snake.Cobra
}
public sealed class Snake permits Snake.Cobra {
   final class Cobra extends Snake {}     // OK
}

Sealed types and pattern-matching switch

Like enums, a sealed type lets a switch be exhaustive without a default — but only if the compiler can see every possibility. A sealed abstract parent guarantees that.

abstract sealed class Fish permits Trout, Bass {}
final class Trout extends Fish {}
final class Bass  extends Fish {}

String getType(Fish fish) {
   return switch (fish) {
      case Trout t -> "Trout!";
      case Bass  b -> "Bass!";
   };  // no default needed — but removing 'abstract' from Fish breaks this
}
SECTION 04

Encapsulation & Records

Encapsulation protects class members by making instance variables private and forcing access through methods. A class is well-encapsulated as long as its instance variables are private — getters and setters are optional. Omit setters to make it immutable.

Records remove the boilerplate

A record is a compact, immutable, encapsulated data carrier. The compiler generates a constructor, an accessor per field, and equals(), hashCode(), toString().

public record Crane(int numberEggs, String name) {}

var mommy = new Crane(4, "Cammy");
System.out.println(mommy.numberEggs());   // 4   (accessor is name(), NOT getName())
System.out.println(mommy.name());         // Cammy
System.out.println(mommy);                // Crane[numberEggs=4, name=Cammy]
Key Rule — generated members

Constructor parameters follow the declaration order. Accessors are the bare field name plus () — no get/is prefix. Two records with equal fields are equals() and share a hashCode(). A field-less record is legal: public record Crane() {}.

var m1 = new Crane("Cammy", 4);  // DOES NOT COMPILE — wrong order
var m2 = new Crane("Cammy");     // DOES NOT COMPILE — missing argument

Constructors: long, compact, overloaded

Compact constructor

  • Declared with no parentheses: public Crane {}
  • Validates / transforms the parameters.
  • Java runs the real constructor afterward.
  • Cannot assign fields via this.x.

Long & overloaded

  • Long: same params as declaration; must set every field.
  • Overloaded: different params; first line must be this(...).
  • Only the long form may set fields with this.x.
// Compact — transform the parameter, not the field:
public record Crane(int numberEggs, String name) {
   public Crane {                          // no parentheses!
      if (name == null || name.length() < 1)
         throw new IllegalArgumentException();
      name = name.substring(0,1).toUpperCase() + name.substring(1).toLowerCase();
      this.numberEggs = 10;                // DOES NOT COMPILE — can't set field in compact ctor
   }
}

// Empty long constructor leaves fields unset:
public record Crane(int numberEggs, String name) {
   public Crane(int numberEggs, String name) {}   // DOES NOT COMPILE — fields not assigned
}
Exam Trap — record members

Records are implicitly final, can't extend anything, and forbid instance fields outside the header and instance initializers. Static fields and static initializers are allowed.

public record Crane(int numberEggs, String name) {
   private static int TYPE = 10;            // OK — static field
   public int size;                          // DOES NOT COMPILE — instance field
   static { System.out.print("Hi"); }        // OK — static initializer
   { System.out.print("Bye"); }              // DOES NOT COMPILE — instance initializer
}
public record BlueCrane() extends Crane {}  // DOES NOT COMPILE — records are final

Record pattern matching (new in Java 21)

Record deconstruction rules

  • If any field is included, all fields must be included, in order.
  • Field names may differ from the record's.
  • Each pattern type must be compatible with the declared type (no numeric promotion).
  • You can name the record or its fields — not both.
record Monkey(String name, int age) {}

if (animal instanceof Monkey m) {}                          // OK — simple binding
if (animal instanceof Monkey(String n, int a) m) {}        // DOES NOT COMPILE — named record AND fields
if (animal instanceof Monkey(String n, long a)) {}         // DOES NOT COMPILE — numeric promotion
if (animal instanceof Monkey(Object n, int a)) {}         // OK — String is compatible with Object

SWITCH ON A RECORD — showData(Snake)

  • record Snake(Object data) {} matched by Long, Integer, Number, Object cases.
  • new Snake(1) → 11  ·  new Snake(2L) → 3  ·  new Snake(3.0) → 103
SECTION 05

Creating Nested Classes

A nested class is defined inside another class. There are four flavors:

TypeWhereNeeds outer instance?Access modifiers
Inner classmember level, non-staticYesall four
static nested classmember level, staticNoall four
Local classinside a method/blockn/anone
Anonymous classdeclared & instantiated inlinen/anone

Inner classes require an instance

An inner class can use any access level, can access all outer members (even private), and must be created from an instance of the outer class — using the odd outer.new Inner() syntax from a static context.

public class Fox {
   private class Den {}
   public void goHome() { new Den(); }            // OK — instance context
   public static void visitFriend() { new Den(); }  // DOES NOT COMPILE — static, no Fox instance
}
public class Squirrel {
   public void visitFox() { new Den(); }         // DOES NOT COMPILE — not in Fox, and Den is private
}

// Creating an inner instance from outside / static:
var home = new Home();
Room room = home.new Room();        // or: new Home().new Room();

To disambiguate same-named fields across nesting levels, use EnclosingClass.this.field: A.this.x, B.this.x.

static nested classes & nested records

A static nested class is created without an outer instance but cannot reach the outer instance members. Nested records are implicitly static, so they behave the same way.

class Emu1 {
   String name = "Emmy";
   static Feathers createFeathers() { return new Feathers("grey"); }  // OK — record is static
   record Feathers(String color) {
      void fly() { System.out.print(name + " is flying"); }       // DOES NOT COMPILE — outer instance var
   }
}

Local and anonymous classes

Key Rule

A local class can capture only final or effectively final local variables. An anonymous class must extend exactly one class or implement exactly one interface — never both — and its declaration ends with a semicolon.

public void processData() {
   final int length = 5;
   int width = 10;
   class VolumeCalculator {
      public int multiply() { return length * width; }  // DOES NOT COMPILE — width not effectively final
   }
   width = 2;                                          // this reassignment breaks it
}

// Anonymous class — note the trailing semicolon:
SaleTodayOnly sale = new SaleTodayOnly() {
   int dollarsOff() { return 3; }
};
SECTION 06

Understanding Polymorphism

Polymorphism lets one object be accessed through a reference of its own type, a supertype, or an implemented interface — no cast needed when moving up to a supertype.

public class Lemur extends Primate implements HasTail {
   public boolean isTailStriped() { return false; }
   public int age = 10;
   public static void main(String[] args) {
      Lemur lemur = new Lemur();
      System.out.println(lemur.age);            // 10
      HasTail hasTail = lemur;
      System.out.println(hasTail.isTailStriped()); // false
      Primate primate = lemur;
      System.out.println(primate.hasHair());    // true
   }
}
Key Rule — object vs reference

The object type determines which properties exist in memory; the reference type determines which members you can call. Reassigning to a supertype hides members but never destroys them. hasTail.age and primate.isTailStriped() do not compile because those members aren't visible through that reference.

Casting rules

Four casting rules

  1. Subtype → supertype: no explicit cast.
  2. Supertype → subtype: explicit cast required.
  3. An invalid cast to an incompatible type throws ClassCastException at runtime.
  4. The compiler disallows casts between unrelated class types.
Lemur lemur3 = primate;            // DOES NOT COMPILE — missing explicit cast
Bird bird = (Bird) fish;           // DOES NOT COMPILE — Fish and Bird unrelated
Exam Trap — interface casts

The compiler permits a cast to an unrelated interface (a subclass might implement it), so it compiles but throws ClassCastException at runtime. The exception: if the source class is final, the compiler knows no subclass can implement the interface, so it fails to compile. instanceof with unrelated types does not compile either.

Overriding vs hiding

Overriding an instance method replaces it everywhere — calls dispatch on the object's actual type. Hiding (static methods and all variables) is resolved by the reference type.

Overriding (instance method)

  • EmperorPenguin.getHeight() = 8; printInfo() in parent calls this.getHeight() → 8
  • Dispatched on the object type.

Hiding (static / variable)

  • Static getHeight() hidden → parent's printInfo() prints 3
  • Resolved by reference type, not object.
class Marsupial {
   protected int age = 2;
   public static boolean isBiped() { return false; }
}
public class Kangaroo extends Marsupial {
   protected int age = 6;
   public static boolean isBiped() { return true; }
   public static void main(String[] args) {
      Kangaroo joey = new Kangaroo();
      Marsupial moey = joey;
      System.out.println(joey.isBiped());  // true   (reference type Kangaroo)
      System.out.println(moey.isBiped());  // false  (reference type Marsupial)
      System.out.println(joey.age);        // 6      (variable hidden — ref type)
      System.out.println(moey.age);        // 2      (variable hidden — ref type)
   }
}
Remember

You can't reach a grandparent's method through super.super. super.getHeight() inside Penguin.printInfo() refers to Object and does not compile — override printInfo() in the child and call super.getHeight() there to print the parent's value.

Short Recap

  • A class implements many interfaces but extends one class; an interface extends many interfaces. Mixing the keywords across class/interface fails.
  • Interface members are implicitly public; variables are public static final; bodyless methods are abstract. default/static/private methods need bodies; static methods aren't inherited (use the interface name).
  • Enums: values first + semicolon; constructors implicitly private and run once; can implement interfaces but can't be extended; valueOf is exact-match.
  • Sealed: direct subclasses must be final, sealed, or non-sealed; same package; permits optional when in the same file or nested.
  • Records: implicitly final; compact constructors transform parameters (no this.field); no instance fields/initializers outside the header; accessors are name().
  • Nested: inner needs an outer instance; static nested and nested records don't; local classes capture only effectively-final locals; anonymous classes extend one class or one interface.
  • Polymorphism: object type owns the data, reference type controls access; instance methods override (object), statics and variables hide (reference).

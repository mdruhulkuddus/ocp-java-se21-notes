SECTION 01

Collection APIs & Generics Basics

The Java Collections Framework (in java.util) has four main interfaces. Map does not implement Collection — it's part of the framework but keeps key/value methods of its own.

InterfaceMeaning
Listordered, allows duplicates, accessed by int index
Setno duplicate entries
Queue / Dequeorders elements for processing; Deque allows both ends
Mapmaps keys to values; no duplicate keys

Generics shorthand: diamond and var

Generics are parameterized types. The diamond operator <> infers from the left and is allowed only on the right of an assignment; var infers from the right. Don't specify the type on both sides.

List<Integer> list = new ArrayList<>();        // diamond infers from left
var list = new ArrayList<Integer>();            // var infers from right

List<> bad = new ArrayList<Integer>();          // DOES NOT COMPILE (diamond on left)
var map = new HashMap<>();                       // compiles -> HashMap<Object,Object>
Exam Trap — var + diamond together

var map = new HashMap<>(); compiles, but with nothing to infer from, the type becomes HashMap<Object, Object>.

Common Collection methods

MethodBehaviour
boolean add(E e)List always returns true; Set returns false on a duplicate
boolean remove(Object o)removes one match, returns whether something was removed
boolean isEmpty(), int size()count of elements
void clear()discards all elements
boolean contains(Object o)uses equals() on elements
boolean removeIf(Predicate<? super E>)removes all matching elements
void forEach(Consumer<? super T>)iterates without a loop
 
Collection<String> set = new HashSet<>();
set.add("Sparrow");   // true
set.add("Sparrow");   // false (duplicate in a Set)
set.removeIf(String::isEmpty);   // s -> s.isEmpty()

equals() across collections

order sensitivity

  • List.of(1,2).equals(List.of(2,1)) → false (a List cares about order)
  • Set.of(1,2).equals(Set.of(2,1)) → true (a Set ignores order)
  • List.of(1,2).equals(Set.of(1,2)) → false (different types)
Exam Trap — unboxing null

Adding null to a List<Integer> is legal, but unboxing it to a primitive throws: int h = heights.get(0); → NullPointerException.

SECTION 02

The List Interface

A List is ordered and allows duplicates, with index-based access. Two implementations to know:

ArrayList

  • Resizable array; constant-time lookup by index.
  • Best when you read as often as (or more than) you write.
  • The default "when unsure" choice.

LinkedList

  • Implements both List and Deque.
  • Constant-time add/remove/access at the ends.
  • Arbitrary index access is linear time.

Factory methods

MethodResultReplace?Add/Delete?
Arrays.asList(varargs)fixed-size list backed by the arrayYesNo
List.of(varargs)immutable listNoNo
List.copyOf(collection)immutable copyNoNo
Key Rule — backed array vs immutable

Arrays.asList() is backed by the array: changing the array changes the list and set() changes the array back. List.of() and List.copyOf() are immutable — any set(), add(), or remove() throws UnsupportedOperationException. (And Arrays.asList().add() also throws, since the size is fixed.)

List methods and the overloaded remove() trap

Beyond the inherited Collection methods: add(int, E), get(int), indexOf, lastIndexOf (both return -1 if not found), remove(int), set(int, E), replaceAll(UnaryOperator), and sort(Comparator). set() on an out-of-range index throws IndexOutOfBoundsException.

var list = new LinkedList<Integer>();
list.add(3); list.add(2); list.add(1);   // [3, 2, 1]
list.remove(2);                  // removes INDEX 2 -> [3, 2]
list.remove(Integer.valueOf(2));  // removes VALUE 2 -> [3]
Exam Trap — remove(int) vs remove(Object)

With an Integer list, remove(int) deletes by index and throws IndexOutOfBoundsException if missing; remove(Object) deletes by value and returns false if missing. remove(Integer.valueOf(2)) forces the value version.

var nums = Arrays.asList(1, 2, 3);
nums.replaceAll(x -> x*2);   // [2, 4, 6]

String[] arr = list.toArray(new String[0]);  // typed array; size 0 lets Java size it

toArray() with no argument returns Object[]. The returned array is a copy — clearing the list does not affect it.

SECTION 03

Set, Queue & Deque

Set implementations

ClassStorage / order
HashSethash table; uses hashCode(); iteration order is arbitrary
LinkedHashSethash table + insertion (encounter) order
TreeSetsorted tree; always sorted; slower add/remove

same elements added 66, 10, 8

  • HashSet → 66,8,10 (arbitrary)
  • LinkedHashSet → 66,10,8 (insertion)
  • TreeSet → 8,10,66 (sorted)

Immutable sets come from Set.of(...) and Set.copyOf(...). Equality uses equals(); hashCode() chooses the bucket.

Queue and Deque

A Queue is FIFO. LinkedList and ArrayDeque both implement Deque (which extends Queue). Use ArrayDeque when you don't need List methods.

Two flavours of each operation

Each capability has a version that throws on failure and one that returns a sentinel (null/false):

  • Add to back: add(e) (throws) · offer(e) (returns boolean)
  • Read front: element() (throws) · peek() (returns null)
  • Remove front: remove() (throws) · poll() (returns null)
Deque purposeFront methodsBack methods
AddaddFirst / offerFirstaddLast / offerLast
Read (peek)getFirst / peekFirstgetLast / peekLast
RemoveremoveFirst / pollFirstremoveLast / pollLast
Remember — Deque as a stack (LIFO)

Used as a stack, the methods are push(e) (add to front/top), pop() (remove front/top), and peek(). A FIFO queue is a line of people; a LIFO stack is a stack of plates.

SECTION 04

The Map Interface

A Map stores key/value pairs and does not extend Collection. Generics are K (key) and V (value).

ClassOrder
HashMapuses hashCode() of keys; arbitrary order
LinkedHashMapinsertion (encounter) order
TreeMapsorted by keys; no null keys
 
Map.of("key1", "value1", "key2", "value2");     // up to 10 pairs
Map.ofEntries(Map.entry("k1", "v1"), Map.entry("k2", "v2"));  // safer
Map.copyOf(map);

Map methods to know

MethodBehaviour
V get(Object key)value or null if absent
V getOrDefault(key, default)value or the default if absent
V put(K, V)adds/replaces; returns previous value or null
V putIfAbsent(K, V)sets only if key absent or mapped to null
V replace(K, V) / replaceAll(BiFunction)changes value(s); skips a missing key
V merge(K, V, BiFunction)combines values (see rules below)
containsKey / containsValuekeys and values checked separately

Exam Trap — contains() is not on Map

map.contains("lion") does not compile — contains() is a Collection method. Maps have containsKey() and containsValue().

merge() behaviour

What merge(key, value, fn) does

  • Key absent, or mapped to null → set to value (function not called).
  • Key has a non-null value → set to fn(old, value).
  • If the function returns null → the key is removed.

longest-name mapper

  • favorites.put("Jenny","Bus Tour"); merge("Jenny","Skyride", longest) → keeps Bus Tour (longer).
  • favorites.put("Tom","Tram"); merge("Tom","Skyride", longest) → Skyride (longer).

forEach on a Map takes two parameters: map.forEach((k, v) -> ...). entrySet() yields Map.Entry<K,V> objects with getKey()/getValue().

SECTION 05

Sorting, Searching & Sequenced Collections

Comparable vs Comparator

DifferenceComparableComparator
Packagejava.langjava.util
Implemented by the class being compared?YesNo
Method namecompareTo(T)compare(T, T)
# parameters12

Key Rule — compareTo() return value

Return 0 when equal, a negative number when this is smaller, a positive number when larger. id - d.id sorts ascending; d.id - id sorts descending.

Exam Trap — wrong method name

A Comparator must implement compare(), not compareTo(). new Comparator<Duck>(){ public int compareTo(...){} } does not compile. Both are functional interfaces (one abstract method), but using a lambda for Comparable would be pointless.

Building a Comparator

Comparator<Duck> byWeight = Comparator.comparing(Duck::weight);
Comparator<Squirrel> c = Comparator.comparing(Squirrel::species)
                                   .thenComparingInt(Squirrel::weight);
var desc = Comparator.comparing(Squirrel::species).reversed();

Static builders: comparing, comparingInt/Double/Long, naturalOrder(), reverseOrder(). Chainable default methods: reversed(), thenComparing, thenComparingInt/Double/Long.

Sorting and binarySearch

Exam Trap — sorting needs Comparable

Collections.sort(rabbits) on a non-Comparable type does not compile. But adding a non-Comparable element to a TreeSet compiles and throws ClassCastException at runtime (on the first add). Pass a Comparator to fix either case.

List<Integer> list = Arrays.asList(6,9,1,8);
Collections.sort(list);                       // [1, 6, 8, 9]
Collections.binarySearch(list, 6);            // 1  (found at index)
Collections.binarySearch(list, 3);            // -2  (-(insertion point)-1)

binarySearch requires a sorted list; a found element returns its index, otherwise -(insertion point) - 1. Searching with a comparator whose order doesn't match the list's order is undefined.

Sequenced collections (Java 21)

SequencedCollection, SequencedSet, and SequencedMap have a well-defined encounter order (not necessarily sorted). Methods include addFirst/addLast, getFirst/getLast, removeFirst/removeLast, and reversed(); maps add firstEntry/lastEntry, pollFirstEntry/pollLastEntry, putFirst/putLast.

Remember — Hash types are not sequenced

HashSet and HashMap are the notable exceptions with no encounter order, so a HashMap can't be a SequencedMap (won't compile). On a TreeSet, addLast() throws UnsupportedOperationException — you can't force position in a sorted structure. Sorted sets forbid null values; sorted maps forbid null keys.

An unmodifiable view (Collections.unmodifiableList/Set/Map/Collection) can't be changed through the view, but changes to the underlying collection still show through it.

SECTION 06

Generics & Wildcards

Declare a formal type parameter in angle brackets. Convention: E element, K key, V value, N number, T type, S/U/V for more.

public class Crate<T> {
   private T contents;
   public T lookInCrate() { return contents; }
   public void packCrate(T contents) { this.contents = contents; }
}
public record CrateRecord<T>(T contents) {}   // generic record

Type erasure

At compile time the compiler replaces each T with Object (or its bound) and inserts casts. There is only one class file. This has consequences for overloading and overriding.

Exam Trap — erasure breaks overloading

You can't have both chew(List<Object>) and chew(List<Double>) — after erasure both become chew(List) → does not compile. Same across parent/child. But chew(List<Object>) and chew(ArrayList<Double>) coexist (different raw types).

Exam Trap — covariant returns, exact generics

An overriding method's return type may be a subtype, but the generic parameter must match exactly. ArrayList<CharSequence> play() overriding List<CharSequence> play() compiles; List<String> play() does not (String ≠ CharSequence), even though String sleep() overriding CharSequence sleep() is fine.

Generic methods

public static <T> void prepare(T t) { }
public static <T> Crate<T> ship(T t) { return new Crate<T>(); }
public static T noGood(T t) { return t; }   // DOES NOT COMPILE (missing <T>)

The formal type parameter <T> goes immediately before the return type (unless inherited from the class). A method's T is independent of the class's T.

Wildcards

BoundSyntaxExample
Unbounded?List<?> a = new ArrayList<String>();
Upper bound? extends typeList<? extends Number> a = new ArrayList<Integer>();
Lower bound? super typeList<? super Exception> a = new ArrayList<Object>();
Key Rule — upper/unbounded are read-only

List<String> can't be assigned to List<Object> (type safety). Use List<?> for "a list of anything." With an unbounded or upper-bounded wildcard the list is logically immutable: birds.add(new Sparrow()) on a List<? extends Bird> does not compile. Upper bounds use extends for both classes and interfaces.

Key Rule — lower bound lets you add

List<? super String> accepts a List<String> or any supertype list, so adding a String is safe. For List<? super IOException> you can add an IOException or its subclass FileNotFoundException, but not a plain Exception (it might really be a List<IOException>).

List<? extends B> l4 = new ArrayList<A>();    // DOES NOT COMPILE (A is not B/sub)
List<?> l6 = new ArrayList<? extends A>();    // DOES NOT COMPILE (can't instantiate a wildcard)
<B extends A> B third(List<B> list){ return new B(); }  // DOES NOT COMPILE (can't new a type param)
<X> void fifth(List<X super B> list){ }           // DOES NOT COMPILE (wildcard needs ?)
Remember — what you can't do with generics

Because of erasure you can't: call new T(), create new T[], use instanceof on a parameterized type, use a primitive as a type argument, declare a static field of the type parameter, or catch a T.

Short Recap

  • Four interfaces: List (ordered, dups), Set (no dups), Queue/Deque (processing order), Map (key/value, not a Collection).
  • Diamond <> infers from the left; var from the right; var x = new HashMap<>() gives <Object,Object>.
  • Arrays.asList = fixed-size, backed array (set ok, add/remove throw); List.of/copyOf = fully immutable.
  • remove(int) is by index (throws if missing); remove(Object) is by value (returns false). Integer.valueOf(n) forces the value version.
  • HashSet/HashMap unordered, Linked* insertion order, Tree* sorted (no null keys/values).
  • Queue/Deque: throwing vs sentinel pairs (add/offer, element/peek, remove/poll); stack uses push/pop/peek.
  • merge: function runs only with two real values; returns null → remove. putIfAbsent updates a null mapping. map.contains() doesn't compile.
  • Comparable: compareTo, 1 param, java.lang, in the class. Comparator: compare, 2 params, java.util, usually a lambda.
  • Collections.sort on a non-Comparable doesn't compile; a TreeSet add throws ClassCastException at runtime.
  • Type erasure replaces T with Object; blocks overloading by generic type; requires exact generic match for covariant overrides.
  • Wildcards: ? / ? extends (read-only) / ? super (can add). Generic methods declare <T> before the return type.

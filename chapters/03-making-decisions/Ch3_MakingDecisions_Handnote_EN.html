SECTION 01

Decision-Making: if / else & Pattern Matching

An if statement runs a block only when a boolean expression is true. An else (and chained else if) handles the alternatives. Java evaluates branches top to bottom and stops at the first true.

⚠ EXAM TRAP — indentation lies; trace the braces 
if (hourOfDay < 11)
  System.out.println("Good Morning");
  morningGreetingCount++;  // runs ALWAYS — not part of the if

Without braces, an if governs only the next single statement. Tabs are just whitespace in Java. Always trace the open/close braces, ignoring indentation.

⚠ EXAM TRAP — if needs a boolean 
int hourOfDay = 1;
if (hourOfDay) { ... }  // DOES NOT COMPILE (0 and 1 are not boolean)

Pattern matching with instanceof

Pattern matching adds a pattern variable after the type, combining the type check and the cast in one step — and the cast runs only when instanceof is true, avoiding ClassCastException.

// old, verbose
if (number instanceof Integer) {
  Integer data = (Integer) number;
  System.out.print(data.compareTo(5));
}
// pattern matching
if (number instanceof Integer data)
  System.out.print(data.compareTo(5));

An optional guard (a boolean expression with &&) can filter further, using the pattern variable on the same line it's declared:

if (number instanceof Integer data && data.compareTo(5) > 0)
  System.out.print(data);
✓ KEY RULE — null & reassignment

instanceof (and pattern matching) is always false for null. Reassigning a pattern variable is bad practice; with final it won't compile (if (n instanceof final Integer data) { data = 10; } → does not compile). The pattern type must be the same, a subtype, or a supertype — an impossible type like Number n instanceof String s does not compile.

Flow scoping

A pattern variable is in scope only where the compiler can prove its type — this is unlike ordinary hierarchical scoping.

⚠ EXAM TRAP — OR breaks scope 
if (number instanceof Integer data || data.compareTo(5) > 0)  // DOES NOT COMPILE

With ||, if the input isn't an Integer, data is undefined — so it can't be used. (With && it's fine.)

But a pattern variable can escape the if when flow guarantees the type. This compiles, because reaching the last line proves number is an Integer:

void printOnlyIntegers(Number number) {
  if (!(number instanceof Integer data))
    return;
  System.out.print(data.intValue());  // in scope here
}
SECTION 02

switch Statements

A switch evaluates one value and redirects flow to one or more branches. A statement doesn't return a value (unlike an expression). Both start with switch (var) and support zero or more case clauses; each clause lists comma-separated values, a separator (: or ->), then an expression or block.

Supported switch types

★ REMEMBER — allowed switch variable types

int/Integer, byte/Byte, short/Short, char/Character, String, enum values, any object type (with pattern matching), and var if it resolves to one of these. Not supported: boolean, long, float, double.

Acceptable case values

case values must be compile-time constants: literals, enum constants, or final variables initialized with a literal. Methods and non-final variables don't qualify.

final int bananas = 1;
int apples = 2;
final int cookies = getCookies();
switch (numberOfAnimals) {
  case bananas:        // OK (final + literal)
  case apples:         // DOES NOT COMPILE (not final)
  case getCookies():   // DOES NOT COMPILE (method)
  case cookies:        // DOES NOT COMPILE (final from a method, not a literal)
  case 3 * 5:         // OK (constant expression)
}

The case type must match the switch variable's type. A String case on an int switch does not compile. If -> and : are mixed across clauses, it does not compile — pick one style.

Fall-through & break

⚠ EXAM TRAP — colon-style falls through

Without break, execution continues into the following branches in order, including default wherever it sits. printSeasonForMonth(2) below prints everything:

switch (month) {
  case 1, 2, 3: System.out.print("Winter-");
  case 4, 5, 6: System.out.print("Spring-");
  default:     System.out.print("Unknown-");
  case 7, 8, 9: System.out.print("Summer-");
  case 10,11,12: System.out.print("Fall-");
}
// Winter-Spring-Unknown-Summer-Fall-

Order matters! Add break after each branch to stop fall-through. A switch with no case clauses (switch (month) {}) is valid.

★ REMEMBER — arrow-style switch statement

An arrow switch whose bodies return void (e.g. all System.out.print(...)) is still a switch statement, not an expression — it doesn't return a value. Arrow clauses don't fall through.

SECTION 03

switch Expressions

A switch expression returns a value, so it usually sits on the right of = or after return, and needs a trailing semicolon. Each arrow case expression also needs its own semicolon; a case block (braces) does not.

var result = switch (bear) {
  case 30 -> "Grizzly";   // semicolon after each case expression
  default -> "Panda";
};                          // semicolon after the whole expression

Consistent return type

Every branch must return a value compatible with the target type.

int size = switch (measurement) {
  case 5  -> Integer.valueOf(1); // OK (unboxed to int)
  case 10 -> (short)2;         // OK (short fits int)
  default -> 3;
  case 20 -> "4";             // DOES NOT COMPILE
  case 40 -> 5L;             // DOES NOT COMPILE
  case 50 -> null;           // DOES NOT COMPILE
};

Exhaustiveness

A switch expression must be exhaustive — three ways

  1. Add a default clause (most common).
  2. For an enum, add a case for every value.
  3. Cover all possible types via pattern matching.

An enum switch missing one value (e.g. FALL) does not compile until you add that case or a default. Writing a case for every int (or even every byte) is not accepted — you need default.

The yield statement

A case block returns its value with yield (a return would exit the whole method). yield is required in a block unless it throws.

var name = switch (fish) {
  case 1 -> "Goldfish";
  case 2 -> { yield "Trout"; }
  case 3 -> {
    if (length > 10) yield "Blobfish";
    else yield "Green";
  }
  default -> "Swordfish";
};
⚠ EXAM TRAP — semicolons with case blocks

A case expression needs a semicolon; a case block must not have one after the closing brace. case 2 -> { yield "Trout"; }; has an extra semicolon and does not compile.

Pattern matching switch (Java 21)

Use a type + pattern variable in each case. A when guard adds a condition (the keyword when is required). This lets switch handle ranges.

return switch (height) {
  case Integer i when i > 10 -> "Joseph";
  case Integer i              -> "Daniel";
  case Double num when num <= 15.5 -> "Peter";
  case Double num             -> "Kelly";
  case Number num             -> "Ralph";
};
⚠ EXAM TRAP — dominance & ordering

With pattern matching, order matters for both statements and expressions. A broader case before a narrower one makes the narrower unreachable → compile error. case Integer i before case Integer i when i > 10 does not compile. A type unrelated to the switch variable (Number switch, case String s) does not compile.

✓ KEY RULE — case null & runtime null

Without case null, a null switch variable throws NullPointerException at runtime (even with a default). Java 21 allows case null. Using case null makes the switch a pattern-matching switch (so it must be exhaustive), and case null cannot come after default.

SECTION 04

Loops: while, do-while, for, for-each

while & do-while

A while checks its condition before each iteration, so the body may run zero times. A do/while runs the body at least once, then checks.

while (cond) { ... }

  • Condition first.
  • Zero or more iterations.
  • while (full < 5) with full=5 prints nothing.

do { ... } while (cond);

  • Body first, condition after.
  • One or more iterations.
  • do { lizard++; } while (false); → lizard is 1.
⚠ EXAM TRAP — infinite loops

If no variable in the condition changes, the loop never ends: while (pen < 10) pigs++; with pen fixed loops forever. Make sure the loop variable moves toward termination.

The basic for loop

Three semicolon-separated sections: init; condition; update. Init and update may have multiple comma-separated statements; declared variables must share one type.

⚠ EXAM TRAP — for-loop quirks 
for (int i = 0; i < 10; i++) ...;
System.out.println(i);  // DOES NOT COMPILE (i scoped to the loop)

for ( ; ; ) ...           // valid infinite loop (all sections optional)

int x = 0;
for (int x = 4; x < 5; x++) ...  // DOES NOT COMPILE (x redeclared)

for (long y = 0, int z = 4; ...) ... // DOES NOT COMPILE (mixed types)
★ REMEMBER — reading reverse loops

Decrementing loops are heavily tested. for (var c = 4; c >= 0; c--) prints 4 3 2 1 0. Watch the start value and whether the condition uses > or >=.

The for-each loop

The right side must be a built-in array or something that implements java.lang.Iterable (e.g. List, Set — but not Map, and not a String). The left side declares a variable whose type matches each element.

for (var name : names) System.out.println(name);

String birds = "Jay";
for (String bird : birds) ...   // DOES NOT COMPILE (String isn't iterable here)
String[] sloths = new String[3];
for (int sloth : sloths) ...    // DOES NOT COMPILE (int vs String)
SECTION 05

Branching: break, continue, return, labels

A label is an identifier followed by a colon placed before a loop, letting break/continue target an outer loop. By convention labels are SNAKE_CASE.

break

  • Exits the loop entirely.
  • break LABEL; exits the labelled outer loop.

continue

  • Ends the current iteration, re-checks the condition.
  • continue LABEL; continues the labelled loop.

Unlabelled break/continue apply to the nearest inner loop. return exits the whole method and can replace labels for cleaner nested-loop searches.

⚠ EXAM TRAP — unreachable code 
if (checkDate > 100) {
  break;
  checkDate++;  // DOES NOT COMPILE (unreachable)
}

Any statement immediately after break, continue, or return in the same block is unreachable and fails to compile — even if the branch looks logically impossible.

★ REMEMBER — labelled break changes the result

In a nested search, break LABEL; exits all loops at the first match; a plain break; only exits the inner loop (finding the last inner match); no break finds the very last match overall. The output differs in each case.

★ SHORT RECAP

Short Recap

Decisions & pattern matching

  • if needs a boolean (not 0/1); without braces it governs one statement — trace braces, not indentation.
  • Pattern matching merges check + cast; null is always false. Flow scoping keeps the variable in scope only where its type is proven (&& ok, || not; early return can extend scope).

switch

  • Allowed types: int/Integer, byte/Byte, short/Short, char/Character, String, enum, object (pattern), var; never boolean/long/float/double. Cases must be compile-time constants matching the switch type.
  • Colon-style falls through (use break); arrow-style doesn't. Expressions must be exhaustive and return a consistent type; yield returns from a block; case null implies pattern matching and can't follow default.

Loops & branching

  • while = zero-or-more; do/while = one-or-more. for sections are optional but the two semicolons aren't; loop variables are scoped to the loop. for-each needs an array or Iterable (not Map/String).
  • Labels let break/continue target outer loops; return exits the method; code right after break/continue/return is unreachable.
